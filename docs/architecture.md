# Deepiri IDE Desktop — Architecture

High-level architecture of the desktop application and how it fits into the Deepiri platform.

---

## Tech stack (actual)

### Desktop & UI

| Layer | Technology | Role |
|-------|------------|------|
| **Desktop runtime** | Electron 28 | Main process, window management, IPC, native menus |
| **Renderer** | React 18 (JSX) | UI components, state, routing (single-page) |
| **Build** | Vite 5 | Dev server (HMR), production bundle for renderer |
| **Editor** | Monaco Editor (@monaco-editor/react) | Code editing, syntax highlighting, themes |

### Main process (Node.js)

- **Entry:** `src/main.js`
- **Responsibilities:** Window lifecycle, IPC handlers (file system, shell, API proxy, AI/Cyrex/Helox), security (contextBridge via preload).
- **Preload:** `src/preload.js` — exposes a controlled `window.electronAPI` to the renderer.

### Renderer (React)

- **Entry:** `src/renderer/main.jsx` → `App.jsx`
- **Structure:** `components/`, `features/`, `context/`, `hooks/`, `services/`, `styles/`, `integrations/`.
- **State:** React state + context (Theme, Notifications); no global store (Redux) by default.

---

## Architecture layers

### 1. Main process (Electron)

- Creates `BrowserWindow`, loads renderer (dev: Vite URL; prod: `dist-renderer/index.html`).
- IPC: `list-directory`, `create-file`, `create-folder`, `delete-path`, `rename-path`, `get-project-root`, `open-project`, `open-file`, `save-file`, `run-command`, `cancel-command`, `api-request`, `ai-request`, `classify-task`, `generate-challenge`, Helox pipeline run, etc.
- No direct DOM access; all file/shell access goes through main.

### 2. Renderer (React)

- **Activity bar + sidebar:** Explorer (workspace tree or legacy tasks), Tasks, Challenges, Gamification, Integrations, Cyrex, Pipelines, Search, Outline, Keybindings, Extensions, AI panel.
- **Editor area:** Tabs, Monaco editor, breadcrumbs; optional Cyrex embed, Pipelines view, Search panel.
- **Bottom panel:** Terminal, Output, Problems.
- **Overlays:** Command palette (Ctrl+Shift+P), Quick Open (Ctrl+P), Notifications, Diff view (when used).

### 3. Optional backend services

- **Platform API** (`API_URL`): Tasks, sessions, gamification.
- **Cyrex AI** (`AI_SERVICE_URL`): Classification, challenges, RAG, agents.
- **Cyrex UI** (`CYREX_INTERFACE_URL`): Embedded iframe in Cyrex tab.
- **Helox** (`HELOX_PATH`): Training pipelines run via main process (subprocess/scripts).

---

## Key features (implemented)

- **Workspace:** Open folder → real file tree; create/rename/delete files and folders; open files in tabs.
- **Monaco editor:** Syntax highlighting, themes (ThemeContext), save (Ctrl+S), cursor/selection for AI context.
- **AI:** Context-aware chat (current file + selection), “Apply to file”; legacy AIAssistant when no file open.
- **Quick Open / Command palette:** Ctrl+P file picker, Ctrl+Shift+P commands.
- **Welcome:** Recent folders, quick actions, link to docs/features.
- **Terminal:** Integrated panel, cwd = project root, streamed output.
- **Cyrex & Helox:** Tabs and IPC to backend/pipelines; optional services.

---

## Security and packaging

- **Context isolation:** Preload script only exposes whitelisted API; no `nodeIntegration` in renderer.
- **Build:** `electron-builder` produces installers (NSIS, DMG/PKG, deb/AppImage); `dist-renderer/` is generated by Vite and included in the app package.
